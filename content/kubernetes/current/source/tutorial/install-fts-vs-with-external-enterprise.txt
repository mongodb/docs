.. _k8s-fts-vs-install-external-enterprise:

==========================================================================================
Install and Use |text-search| and |vector-search| With External MongoDB Enterprise Edition
==========================================================================================

.. meta::
   :description: Install MongoDB Search and Vector Search using the MongoDB Controllers for Kubernetes Operator to run with an external MongoDB Enterprise Edition replica set.

.. default-domain:: mongodb

.. contents:: On this page
   :local:
   :backlinks: none
   :depth: 2
   :class: singlecol

You can use the |k8s-op-short| to deploy |text-search| and
|vector-search| on a |k8s| cluster to run with an external
MongoDB Enterprise Edition 8.2 or later server. This tutorial
shows you how to deploy and configure one or more ``mongot``
processes in your |k8s| cluster to use a new or existing external
replica set or sharded cluster.

.. composable-tutorial::
   :options: cluster-topology, mongot-instances
   :defaults: repl, single

   Overview
   --------

   .. selected-content::
      :selections: repl, single

      This tutorial shows you how to configure a secure deployment
      with |tls| enabled. You must provide the |tls| certificates:

      - The server certificate and key for the MongoDBSearch service.
      - The public CA certificate for the external MongoDB database.

   .. selected-content::
      :selections: repl, multiple

      This tutorial shows you how to deploy **MongoDB Search** against
      your **existing external MongoDB replica set** (running on VMs,
      another |k8s| cluster, or a self-managed deployment) using the
      operator's **managed Envoy load balancer**. The |k8s-op-short|
      automatically deploys and configures the Envoy proxy. Therefore,
      you don't need to create Envoy ConfigMaps, Deployments, or
      Services.

      .. figure:: /images/search_external_repl_set_traffic_flow.png
         :figwidth: 700px
         :alt: Traffic flow for external replica set with managed Envoy load balancer

   .. selected-content::
      :selections: shard, multiple

      This tutorial shows you how to deploy **MongoDB Search** against
      your **existing external MongoDB sharded cluster** (running on
      VMs, bare metal, or another |k8s| cluster) using the
      operator's **managed Envoy load balancer**.

      The |k8s-op-short| can deploy and manage |text-search| and |vector-search|
      components in your |k8s| cluster. It can also deploy and manage an Envoy
      proxy to load balance traffic between your external MongoDB sharded
      cluster and the ``mongot`` pods.

      About the Managed Envoy Load Balancer
      ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

      Envoy is a high-performance L7 proxy used for traffic routing.
      When you set ``spec.loadBalancer.managed`` in your
      MongoDBSearch resource, the |k8s-op-short| automatically:

      1. **Deploys an Envoy proxy** - A Deployment that handles
         L7 (application layer) load balancing
      2. **Generates routing configuration** - SNI-based routing
         rules for each shard
      3. **Creates proxy Services** - One Kubernetes Service per
         shard for traffic routing
      4. **Manages TLS** - Configures mTLS between mongod, Envoy, and ``mongot``

      You don't need to write Envoy configuration, deploy Envoy, or
      create proxy Services manually.

      .. figure:: /images/search_external_sharded_cluster_traffic_flow.png
         :figwidth: 700px
         :alt: Traffic flow for external sharded cluster with managed Envoy load balancer

      About Per-Shard Routing
      ~~~~~~~~~~~~~~~~~~~~~~~

      In a sharded cluster, each shard has its own data. The
      |k8s-op-short| deploys separate ``mongot`` instances per shard,
      and each shard's ``mongod`` connects to its corresponding
      ``mongot``. :abbr:`SNI (Server Name Indication)` routing allows
      the Envoy proxy to inspect the TLS handshake and route traffic
      to the correct ``mongot`` based on the hostname.

   Prerequisites
   -------------

   .. selected-content::
      :selections: repl, single

      To deploy |text-search| and |vector-search|, you must have the
      following:

      - A running |k8s| cluster with ``kubeconfig`` available locally.
      - |k8s| command-line tool, ``kubectl``, configured to communicate with
        your cluster.
      - |helm|, the package manager for |k8s|, to install the |k8s-op-short|.
      - Bash v5.1 or later for running the commands in this tutorial.
      - A MongoDB Enterprise Edition replica set running version 8.2 or
        later for storing data.

      .. important::

         The version format for the Enterprise edition is
         ``X.Y.Z-ent``. Ensure that you choose ``8.2.0`` or later.
         To learn more about MongoDB versioning, see
         :ref:`release-version-numbers` in the MongoDB Manual.

      To learn more about deploying MongoDB Enterprise, see
      :ref:`mdb-resources`.

      - A running MongoDB |com| for managing MongoDB tasks.

      - External DB |tls|
      
        Your external MongoDB Enterprise deployment must be configured to use
        and require |tls| connections.
      
      - External DB CA file
      
        You must have the Certificate Authority (CA) certificate file
        (for example, ``external-ca.crt``) that was used to sign your external
        MongoDB deployment's server certificates. You need this file
        locally.

      - Search Service certificate files
      
        You must have a valid |tls| server certificate and private key (for
        example, ``search-service.crt``, ``search-service.key``) for the
        MongoDBSearch service. This certificate must be valid for
        the hostname you use to access the service (for example,
        ``mdbs-search.example.com``).

   .. _k8s-search-set-parameters:

   .. collapsible::
      :heading: Configure Parameters for MongoDB Search and Vector Search
      :expanded: false

      Log in to the |com| UI and perform the following steps to configure
      |com| for |text-search| and |vector-search|.

      .. procedure::
         :style: normal

         .. step:: Go to your |com| :guilabel:`Deployment` page.

            a. Log in to MongoDB |com|.
            #. Click the :guilabel:`Deployment` tab.
            #. Select the cluster that you want to modify.

         .. step:: Modify your |com| deployment configuration.

            a. Click the :guilabel:`Modify` button to open the deployment
               configuration editor.
            #. Click :guilabel:`Advanced Configuration Options` under the
               :guilabel:`Process Configuration` section.
            #. Click the :guilabel:`Add Option` button and select
               :guilabel:`setParameter Startup Option` from the dropdown.
            #. Add the following parameters in the fields, one by one, by
               clicking :guilabel:`Add` after adding the name and value:

               .. selected-content::
                  :selections: repl, single

                  .. list-table::
                     :widths: 30 70
                     :header-rows: 1

                     * - Parameter
                       - Value

                     * - ``mongotHost``
                       - Your search hostname and port. For example:
                         ``search-node1.example.com:27028``.

                     * - ``searchIndexManagementHostAndPort``
                       - Your search hostname and port. For example:
                         ``search-node1.example.com:27028``.

                     * - ``skipAuthenticationToSearchIndexManagementServer``
                       - ``false``

                     * - ``searchTLSMode``
                       - Your configured TLS mode. For example, ``requireTLS``,
                         if the ``mongot`` process is configured to accept TLS
                         connections.

                     * - ``useGrpcForSearch``
                       - ``true``

               .. selected-content::
                  :selections: repl, multiple

                  .. list-table::
                     :widths: 30 70
                     :header-rows: 1

                     * - Parameter
                       - Value

                     * - ``mongotHost``
                       - The Envoy proxy service endpoint. Use the
                         hostname configured in
                         ``spec.loadBalancer.managed.externalHostname``.
                         For example:
                         ``{name}-search-0-proxy-svc.{namespace}.svc.cluster.local:27028``.

                     * - ``searchIndexManagementHostAndPort``
                       - The Envoy proxy service endpoint. Use the same
                         value as ``mongotHost``. For example:
                         ``{name}-search-0-proxy-svc.{namespace}.svc.cluster.local:27028``.

                     * - ``skipAuthenticationToSearchIndexManagementServer``
                       - ``false``

                     * - ``searchTLSMode``
                       - ``requireTLS``

                     * - ``useGrpcForSearch``
                       - ``true``

               .. selected-content::
                  :selections: shard, multiple

                  For a sharded cluster, you must set parameters at
                  three levels: cluster-wide, per-shard, and on
                  ``mongos``. Don't set search parameters on the
                  config servers.

                  **Cluster level** (applies to all shards and ``mongos``):

                  .. list-table::
                     :widths: 30 70
                     :header-rows: 1

                     * - Parameter
                       - Value

                     * - ``searchTLSMode``
                       - ``requireTLS``

                     * - ``useGrpcForSearch``
                       - ``true``

                     * - ``skipAuthenticationToMongot``
                       - ``false``

                     * - ``skipAuthenticationToSearchIndexManagementServer``
                       - ``false``

                  **Each shard** (set on shard-0, shard-1, and so on):

                  .. list-table::
                     :widths: 30 70
                     :header-rows: 1

                     * - Parameter
                       - Value

                     * - ``mongotHost``
                       - That shard's Envoy proxy endpoint. For example,
                         for ``shard-0``:
                         ``{name}-search-0-shard-0-proxy-svc.{namespace}.svc.cluster.local:27028``.

                     * - ``searchIndexManagementHostAndPort``
                       - Same value as ``mongotHost`` for this shard.

                  **mongos**

                  .. list-table::
                     :widths: 30 70
                     :header-rows: 1

                     * - Parameter
                       - Value

                     * - ``mongotHost``
                       - Any shard's Envoy proxy endpoint. For example:
                         ``{name}-search-0-shard-0-proxy-svc.{namespace}.svc.cluster.local:27028``.

                     * - ``searchIndexManagementHostAndPort``
                       - Same value as ``mongotHost``.

         .. step:: Review and deploy your changes.

            a. Click the :guilabel:`Review & Deploy` button.
            #. Review and confirm to apply the changes.

               .. selected-content::
                  :selections: repl, single

                  |com| performs a rolling restart of your replica set to

                  :selections: repl, multiple

   .. _k8s-search-create-role-user:

   .. collapsible::
      :heading: Configure a User for MongoDB Search and Vector Search
      :expanded: false

      You must create a user with the :authrole:`searchCoordinator`
      role. In MongoDB versions 8.2 and later,
      :authrole:`searchCoordinator` is a built-in role.

      To create the user and assign them the built-in
      ``searchCoordinator`` role, complete the following steps using
      either the |com| UI or the |mongosh|:

      .. tabs::

         .. tab:: UI
            :tabid: ui

            .. procedure::
               :style: normal

               .. step:: Go to the :guilabel:`Users` page.

                  a. Log in to |com| UI.
                  #. Go to the :guilabel:`Security` tab and click
                     :guilabel:`Users`.

               .. step:: Click :guilabel:`Add New User`.

               .. step:: Set the credentials for the user.

                  .. list-table::
                     :stub-columns: 1

                     * - :guilabel:`Username`
                       - Enter ``search-sync-source``.

                     * - :guilabel:`Password`
                       - Set a strong, secure password.

                     * - :guilabel:`Authentication Database`
                       - Choose the ``admin`` database.

               .. step:: Assign the ``searchCoordinator`` role to the user.

                  In the :guilabel:`Assign Roles` section, select the built-in
                  ``searchCoordinator`` role from the dropdown for the
                  ``admin`` database.

               .. step:: Click :guilabel:`Add User` to create the user with the ``searchCoordinator`` role.

         .. tab:: mongosh
            :tabid: mongosh

            In |mongosh|, run the following commands:

            .. procedure::
               :style: normal

               .. step:: Switch to the ``admin`` database.

                  .. code-block:: javascript

                     use admin;

               .. step:: Run the following command after replacing the password with a secure password to create the user.

                  .. code-block:: javascript

                     db.createUser({
                        user: "search-sync-source",
                        pwd: "<PASSWORD>", // Replace with your actual password
                        roles: [
                           { role: "searchCoordinator", db: "admin" }
                        ]
                     });

   Troubleshoot
   ~~~~~~~~~~~~

   .. selected-content::
      :selections: repl, single

      ``MongoDBSearch`` Stuck in Pending
      ``````````````````````````````````

      **Symptoms:** The ``MongoDBSearch`` resource stays in ``Pending`` state.

      **Check:**

      .. code-block:: shell

         kubectl describe mongodbsearch ${MDB_SEARCH_RESOURCE_NAME} -n ${MDB_NS}
         kubectl logs -l app.kubernetes.io/component=mongot -n ${MDB_NS} --tail=50

      **Common Causes:**

      - The password Secret for the ``search-sync-source`` user was not
        created.
      - TLS certificates are not ready. Run
        ``kubectl get certificates -n ${MDB_NS}`` to check.
      - The external MongoDB deployment is not reachable from the
        cluster.

      Reading ``mongot`` Logs
      ```````````````````````

      Check ``mongot`` pod logs for connection and readiness state:

      .. code-block:: shell

         kubectl logs ${MDB_SEARCH_RESOURCE_NAME}-search-0 -n ${MDB_NS} --tail=50

      .. list-table::
         :widths: 35 65
         :header-rows: 1

         * - Issue
           - Log Pattern

         * - Bad credentials or missing user
           - ``MongoSecurityException: Exception authenticating``

         * - TLS CA mismatch or expired cert
           - | ``SSLHandshakeException`` 
             | ``PKIX path validation failed``

         * - Unreachable host or DNS failure
           - | ``MongoSocketOpenException`` 
             | ``MongoTimeoutException``

      ``mongot`` logs readiness state initially as follows: 

      .. code-block:: shell 
         :copyable: false

         Not ready, waiting on replication to be initialized...

      When ``mongot`` is ready, it logs the following:

      .. code-block:: shell
         :copyable: false

         Server is ready!

      To increase log verbosity, set ``spec.logLevel: DEBUG`` on the
      ``MongoDBSearch`` CR.

      ``````````````````````

      **Check:**

      .. code-block:: shell

         kubectl describe deployment ${MDB_SEARCH_RESOURCE_NAME}-search-0-lb -n ${MDB_NS}
         kubectl logs -l app=${MDB_SEARCH_RESOURCE_NAME}-search-0-lb -n ${MDB_NS}

      **Common Causes:**

      - TLS certificate Secrets were not found. Ensure that the
        ``mongot`` and load balancer certificates are created before
        the ``MongoDBSearch`` resource.
      - The ConfigMap ``${MDB_SEARCH_RESOURCE_NAME}-search-lb-0-config``
        does not exist. Verify that the ``MongoDBSearch`` resource
        reached ``Running`` state.
      - Image pull errors. Check image pull secrets and registry
        access.

      ``mongod`` Cannot Reach Envoy
      `````````````````````````````

      **Symptoms:** Search queries fail with connection errors on the
      external ``mongod`` instances.

      **Check:**

      .. code-block:: shell

         # Verify that per-shard proxy Services exist
         kubectl get svc -n ${MDB_NS} | grep proxy-svc

         # From your mongod host, test TLS connectivity to the proxy
         openssl s_client -connect <envoy-endpoint>:27028 -servername <sni-hostname>

      **Common Causes:**

      - The ``MongoDBSearch`` resource is not in ``Running`` state.
        Proxy Services are created only after reconciliation completes.
      - Network policies are blocking traffic between the external
        ``mongod`` hosts and the |k8s| cluster.
      - DNS resolution fails for the proxy Service endpoint. Verify
        that the external ``mongod`` hosts can resolve the proxy
        Service's external IP or hostname.
      - The ``mongotHost`` parameter on the external ``mongod`` does
        not match the proxy Service endpoint for that shard.

      Reading ``mongot`` Logs
      ```````````````````````

      Check per-shard ``mongot`` pod logs for connection and readiness
      state:

      .. code-block:: shell

         # Replace <shard-name> with the shard name (for example, shard-0)
         kubectl logs ${MDB_SEARCH_RESOURCE_NAME}-search-0-<shard-name>-0 -n ${MDB_NS} --tail=50

      .. list-table::
         :widths: 35 65
         :header-rows: 1

         * - Issue
           - Log Pattern

         * - Bad credentials or missing user
           - ``MongoSecurityException: Exception authenticating``

         * - TLS CA mismatch or expired cert
           - | ``SSLHandshakeException`` 
             | ``PKIX path validation failed``

         * - Unreachable host or DNS failure
           - | ``MongoSocketOpenException``
             | ``MongoTimeoutException``

      ``mongot`` logs readiness state initially as follows: 

      .. code-block:: shell 
         :copyable: false

         Not ready, waiting on replication to be initialized...

      When ``mongot`` is ready, it logs the following:

      .. code-block:: shell
         :copyable: false

         Server is ready!

      To increase log verbosity, set ``spec.logLevel: DEBUG`` on the
      ``MongoDBSearch`` CR.
