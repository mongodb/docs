.. _create-cluster-instance:

Select {+Cluster+} Tier
-----------------------

Select your preferred {+cluster+} tier. The {+cluster+} tier dictates 
the memory, storage, vCPUs, and |iops| specification for each 
data-bearing server [#data-bearing]_ in the {+cluster+}.

.. note::

   You might see different values depending on your
   selected cloud provider and region.

.. _flex--cluster:

{+Flex-Clusters+}
~~~~~~~~~~~~~~~~~~~

Use {+Flex-clusters+} as an economical way for getting started with MongoDB
and for low-throughput applications. These {+clusters+} deploy to an
environment with access to a subset of |service| features.
To learn more, see :ref:`limits`.

You can deploy one {+Free-cluster+} (free sandbox replica set
cluster) per |service| project. You can :ref:`upgrade <modify-cluster>`
a {+Free-cluster+} to a {+Flex-cluster+} at any time.

{+Flex-clusters+} provide the following added features compared to {+Free-clusters+}:

- :ref:`Backups <flex-snapshots>` for your {+cluster+} data
- Increased storage
- :ref:`API access <graphql-api-overview>`

Considerations
``````````````

{+Flex-clusters+} don't have the full availability of features found in
{+Dedicated-clusters+}. To learn more, see :ref:`limits`. 

.. _atlas-low-traffic-tier:

{+Dedicated-Clusters+} for Low-Traffic Applications
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

``M10`` and ``M20`` {+cluster+} tiers support development environments
and production environments with low-traffic applications.

These clusters support replica set deployments only, but otherwise
provide full access to |service| features.

.. note::

   ``M10`` and ``M20`` {+cluster+} tiers use burstable performance
   infrastructure. Cloud providers cap CPU usage after burst periods,
   which can cause throttling under heavy load. To learn more, see
   :ref:`howitworks-scale-cluster-tier`.

.. _atlas-high-traffic-tier:

{+Dedicated-Clusters+} for High-Traffic Applications
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

``M30`` and higher {+clusters+} are recommended for production
environments.

These clusters support replica set and sharded {+cluster+} deployments 
with full access to |service| features.

Some clusters have variants, denoted by the |angle| character. When you
select these clusters, |service| lists the variants and tags each
{+cluster+} to distinguish their key characteristics.

.. |angle| unicode:: U+276F

.. _aws-gen2-clusters:

{+gen-2-clusters+}
~~~~~~~~~~~~~~~~~~~~~~~~~~~

``M30+`` {+Dedicated-clusters+} on |aws| are available in
:term:`cluster generations <cluster generation>` Gen1 or Gen2.
{+gen-2-clusters+} offer improved performance compared to Gen1 clusters by
supporting {+extended-iops+}, offering the following benefits:

- Independent scaling of your cluster's standard |iops| and storage capacity
- A maximum standard |iops| rate of ``80k`` |iops|
- Disk throughput that scales with your cluster's standard |iops|

When creating a new cluster through the {+atlas-ui+}, {+service+} automatically
selects the :guilabel:`Gen2` option under the :guilabel:`Cluster Tier` section
for eligible clusters. Alternatively, you can select :guilabel:`Gen1` to deploy
a Gen1 Cluster.

.. include:: /includes/cluster-settings/gen2-clusters/aws-gen2-cluster-cli-api-config.rst

For existing clusters, you can switch between generations in the same way you
would change your cluster's tier. For more information, see
:ref:`scale-cluster-instance`.

Compare Gen1 and Gen2 Clusters
``````````````````````````````

The following table lists the differences between Gen1 and Gen2 cluster
generations:

.. list-table::
   :header-rows: 1
   :widths: 40 30 30

   * - 
     - Gen1
     - Gen2

   * - Storage and Standard |iops| Expansion
     - Storage and Standard |iops| scale **together**
     - Storage and Standard |iops| scale **independently**

   * - Maximum Standard |iops|
     - ``16k`` |iops|
     - ``80k`` |iops|

   * - Cluster Tiers
     - All tiers
     - ``M30+`` only

   * - Cloud Providers
     - |aws|, |gcp|, |azure|
     - |aws|

   * - Region Availability
     - All regions for a given cloud provider
     - See :ref:`Gen2 Region Support <aws-reference-gen2-regions>`

   * - Cross-Region Deployments
     - :icon-fa5:`check-circle`
     - Only across |aws| regions that support Gen2 clusters. See :ref:`Gen2 Region Support <aws-reference-gen2-regions>` 

   * - Auto-scaling
     - Reactive and predictive auto-scaling
     - Reactive auto-scaling only

   * - Cross-Cloud Deployment Support
     - :icon-fa5:`check-circle`
     - :icon-fa5:`times-circle`

   * - :ref:`Billing <gen2-aws-clusters-billing>`
     - Storage and |iops| scale together, so your bill reflects scaling for
       both.
     - Storage and |iops| scale independently, so your bill only reflects what
       you scaled.

Sharded Clusters
~~~~~~~~~~~~~~~~

You can use the {+atlas-admin-api+} to choose a different tier per shard
in a sharded cluster. You can also select :term:`Analytics node
<analytics node>` tiers indepenently for each shard. The largest and
smallest shard tiers must be within two tiers of each other. For
example, if the largest shard is ``M50``, the smallest shard can be
``M30`` or ``M40``. If you change the cluster tier for a sharded cluster
in the {+atlas-ui+}, |service| changes the tier of all shards in the
cluster.

You can also use the {+atlas-admin-api+} to choose different |iops| per
shard if the cluster is on |aws| using |aws| provisioned IOPS or the
cluster is on |azure| in regions that support Extended IOPS/storage. 

To learn more, see :ref:`atlas-cluster-sharding` and the
:oas-bump-atlas-op:`Update One Cluster in One Project
<updategroupcluster>` endpoint in the {+atlas-admin-api+} documentation.

Limitations
```````````

Every shard must have an equal disk size on all nodes.
|nvme| {+clusters+} are not compatible with independent shard scaling.

|nvme| instances can't be used in multi-cloud clusters.

.. _nvme-storage:

NVMe Storage
~~~~~~~~~~~~

For applications hosted on :ref:`AWS <amazon-aws>` or  :ref:`Azure <microsoft-azure>`
that require low-latency and high-throughput I/O, |service| offers storage
options using locally attached ephemeral |nvme| |ssd|\s.

.. include:: /includes/fact-nvme-fcbis.rst

.. note::
   
   |service| doesn't support |nvme| {+clusters+} on |gcp|.
   |nvme| {+clusters+} are not compatible with independent shard scaling.

The fixed-value storage space and RAM for an |nvme| {+cluster+}
corresponds to its {+cluster+} tier. To learn more, see 
:ref:`Amazon Cluster Configuration Options
<amazon-aws-configuration-options>` and 
:ref:`Azure Cluster Configuration Options
<microsoft-azure-configuration-options>`.

Clusters with |nvme| storage use :ref:`{+Cloud-Backup+}s 
<backup-cloud-provider>`. You can't disable backup on |nvme| 
clusters. If you want to use hourly backups, |service| limits backups 
on |nvme| clusters to once every 12 hours.

|nvme| clusters use a :manual:`hidden secondary node
</core/replica-set-hidden-member>` that consists of a provisioned 
volume with high throughput and |iops| to facilitate backup.

You can't :ref:`pause <pause-cluster>` an |nvme| {+cluster+}.

NVMe Availability Zones
+++++++++++++++++++++++

|nvme| {+clusters+} in the following |azure| regions have two
:ref:`Availability Zones <microsoft-azure-availability-zones>`:

- ``eastus2``
- ``centralus``
- ``southcentralus``

|nvme| {+clusters+} in all other |azure| regions that 
:ref:`indicate Availability Zones <microsoft-azure-supported-regions>`
have three Availability Zones.

Take the Next Steps
-------------------

You can manage {+clusters+} in the following ways:

.. list-table::
   :widths: 40 60
   :header-rows: 1

   * - Action
     - Description

   * - :ref:`create-cluster-storage`
     - Customize the storage capacity of your {+cluster+}. Each 
       {+cluster+} tier comes with a default set of resources. ``M10+`` 
       {+clusters+} provide the ability to customize your storage 
       capacity.

   * - :ref:`atlas-cluster-sharding`
     - Shard your {+cluster+} to scale horizontally. You can use the 
       |service| Cluster Builder UI, the latest version of the 
       :oas-bump-atlas-tag:`Atlas Admin API <clusters>`, :atlascli:`Atlas CLI </>`,
       or |terraform-atlas-provider| to shard your cluster.

       You can also use the latest version of the :oas-bump-atlas-tag:`Atlas Admin API <clusters>`
       to independently scale each shard in your cluster.

   * - :ref:`cluster-autoscaling`
     - Configure the {+cluster+} tier ranges that |service| uses to
       automatically scale your {+cluster+} tier, storage capacity, or 
       both in response to {+cluster+} usage.

   * - :ref:`cluster-blocking-writes`
     - Learn about write-blocking and how to prevent it. |service| blocks
       writes to your dedicated replica set cluster if the cluster's
       primary node exceeds write-blocking policy thresholds.

   * - :ref:`create-cluster-additional-settings`
     - Configure additional {+cluster+} settings such as MongoDB 
       version, backup, and encryption options.

   * - :ref:`configure-resource-tags`
     - Use resource tags that you provide and manage to categorize 
       resources by purpose, environment, team, or billing center.

   * - :ref:`modify-cluster`
     - Reconfigure an existing {+cluster+}. Modify any of the available 
       |service| configuration options.

   * - :ref:`upgrade-major-MongoDB-version`
     - Manage major version upgrades for your {+cluster+}. |service| 
       enables you to upgrade the major version of an |service| 
       {+cluster+} at any time.

   * - :ref:`configure-maintenance-window`
     - Configure maintenance windows for your {+cluster+}. You can set 
       the hour of the day that |service| should start weekly 
       maintenance on your {+cluster+}.

   * -  :ref:`pause-terminate-cluster`
     - Pause, resume, or terminate an existing {+cluster+}. You can't 
       change the configuration of a paused {+cluster+}. Also, you 
       can't read data from or write data to a paused {+cluster+}.

   * - :ref:`create-cluster-multi-region`
     - Configure multi-cloud distribution for increased availability.
       |service| offers options to improve the availability and 
       workload balancing of your {+cluster+}.

   * - :ref:`replica-set-tags`
     - Use pre-defined replica set tags that |service| provides to 
       direct queries from specific applications to specific node types 
       and regions. To use pre-defined replica set tags in your 
       connection string and direct queries to specific nodes, set the 
       tag in the ``readPreferenceTags`` connection string option.
