========================================================================
Install and Use Search With MongoDB Enterprise Edition
========================================================================

.. meta::
   :description: Install MongoDB Search and Vector Search using the MongoDB Controllers for Kubernetes Operator to run with the MongoDB Enterprise Edition in your Kubernetes cluster.

.. default-domain:: mongodb

.. contents:: On this page
   :local:
   :backlinks: none
   :depth: 2
   :class: singlecol

You can use the |k8s-op-short| to deploy |text-search| and
|vector-search| resources to run with MongoDB enterprise v8.2.0
or later on a |k8s| cluster.

This procedure demonstrates how to deploy
and configure one or more ``mongot`` processes to run with a new or existing
replica set or sharded cluster in your |k8s| cluster.

The deployment uses |tls| certificates to ensure secure communication between
MongoDB nodes and the ``mongot`` search process.

   Overview
   --------

   .. selected-content::
      :selections: repl, multiple

      About the Managed Envoy Load Balancer
      ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

      When you set ``spec.loadBalancer.managed`` in your |fts| resource, the
      operator automatically:

      1. Deploys an Envoy proxy to handle load balancing at application layer.
      2. Generates routing configuration to route traffic to ``mongot`` pods.
      3. Creates a service, a single Kubernetes Service for the Replica Set topology.
      4. Manages |tls| by configuring mTLS between mongod, Envoy, and ``mongot``.
      5. Configures |mongod| by automatically setting search parameters on the Replica Set.

      .. note::

         Envoy uses application-layer (L7) load balancing because ``mongod`` opens a
         single long-lived TCP connection to ``mongot``. An L4 load balancer would
         route all traffic to one ``mongot`` pod, whereas Envoy distributes individual
         HTTP/2 streams across multiple ``mongot`` pods.

      .. important::
         You don't need to write Envoy configuration, deploy Envoy, create proxy
         Services manually, or configure |mongod| search parameters.

      .. image:: /images/mongodb-k8s-replicaset-search-arch.svg
         :alt: Traffic flow diagram showing the Kubernetes cluster with an operator-managed MongoDB replica set (rs-0, rs-1, rs-2), an Envoy proxy load balancer, and ``mongot`` StatefulSet pods.

   .. selected-content::
      :selections: shard, multiple

      This guide walks you through deploying **MongoDB Search** against an
      **operator-managed MongoDB sharded cluster**, deployed through the MongoDB
      Enterprise Kubernetes Operator, using the operator's **managed Envoy load balancer**.

      In this scenario, setting ``spec.source.mongodbResourceRef`` lets the operator automatically
      configure ``mongod`` search parameters, eliminating the need for ``shardOverrides`` or manual
      proxy endpoint configuration.

      About the Managed Envoy Load Balancer
      ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

      When you set ``spec.loadBalancer.managed`` in your |fts| resource, the
      operator automatically:

      1. Deploys an Envoy proxy to handle load balancing at application layer.
      2. Generates routing configuration: |sni| based routing rules for each shard.
      3. Creates proxy Services: One |k8s| service per shard for traffic routing.
      4. Manages |tls| by configuring mTLS between mongod, Envoy, and ``mongot``.
      5. Configures |mongod| by automatically setting search parameters on the Replica Set.

      .. important::
         You don't need to write Envoy configuration, deploy Envoy, create proxy
         Services manually, or configure |mongod| search parameters.


      .. image:: /images/mongodb-k8s-sharded-search-arch.svg
         :alt: Architecture diagram showing the traffic flow from operator-managed shards through Envoy Proxy to ``mongot`` StatefulSets within a Kubernetes cluster.

      About Per-Shard Routing
      ~~~~~~~~~~~~~~~~~~~~~~~

      In a sharded cluster, each shard has its own data. |fts| deploys
      separate ``mongot`` instances per shard, and each shard's |mongod| must connect
      to its corresponding ``mongot``. SNI (Server Name Indication) routing allows the
      Envoy proxy to inspect the TLS handshake and route traffic to the correct
      ``mongot`` based on the hostname.

         Shard names must follow RFC 1123 DNS label rules, which allows 
         lowercase alphanumeric characters or hyphen (``-``) and 
         requires starting with a letter and ending with an alphanumeric 
         character. Underscores are not allowed. The combined length of the 
         resource name and shard name must be less than 50 characters.

   Troubleshooting
   ~~~~~~~~~~~~~~~

   .. selected-content::
      :selections: repl, multiple

      Envoy Pod Not Starting
      ``````````````````````
      **Symptoms:** The ``{name}-search-0-lb`` Deployment has 0/1 ready pods.

      **Check**

      .. code-block:: shell
         :copyable: false

         kubectl describe deployment ${MDB_RESOURCE_NAME}-search-lb-0 -n ${MDB_NS}
         kubectl logs -l app=${MDB_RESOURCE_NAME}-search-lb-0 -n ${MDB_NS}

      **Common causes:**

      - TLS certificate Secrets were not found. Ensure that the ``mongot`` and load 
        balancer certificates are created before the ``MongoDBSearch`` resource.
      - The ConfigMap ``${MDB_SEARCH_RESOURCE_NAME}-search-lb-0-config`` does not 
        exist. Verify that the ``MongoDBSearch`` resource reached Running state.
      - Image pull errors. Check image pull secrets and registry access.

      Search Index Creation Fails
      ````````````````````````````
      **Symptoms:** Search indexes remain in the ``BUILDING`` or ``FAILED``
      state and do not become queryable. Atlas Search queries return errors
      or empty results even after the pods reach the ``Running`` state.

      **Check**

      .. code-block:: shell
         :copyable: false

         kubectl get pods -n ${MDB_NS} | grep search
         kubectl logs ${MDB_RESOURCE_NAME}-search-0 -n ${MDB_NS}

      **Common causes:**

      - ``mongot`` cannot connect to MongoDB. Verify that the source credentials
        are correct and that the ``MongoDBSearch`` resource references a valid secret.
      - TLS CA mismatch between ``mongod`` and ``mongot``. Ensure both components 
        are configured to use the same Certificate Authority.
      - ``mongot`` pods are not ready yet. Wait for all search pods to reach the
        ``Running`` state before attempting to use Atlas Search features.

      MongoDBSearch Stuck in Pending
      ``````````````````````````````

      **Symptoms:** The ``MongoDBSearch`` resource stays in ``Pending`` state.

      **Check**

      .. code-block:: shell
         :copyable: false

         kubectl describe mongodbsearch ${MDB_RESOURCE_NAME} -n ${MDB_NS}
         kubectl get events -n ${MDB_NS} --field-selector involvedObject.name=${MDB_RESOURCE_NAME}

      **Common causes:**

      - The password Secret for the ``search-sync-source`` user was not created.
      - TLS certificates are not ready. Run ``kubectl get certificates -n ${MDB_NS}`` to check.
      - The cluster is not reachable from the Kubernetes cluster.
      ``````````````````````````````````
