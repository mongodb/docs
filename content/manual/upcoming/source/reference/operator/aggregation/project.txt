============================
$project (aggregation stage)
============================

.. facet::
   :name: programming_language
   :values: shell

.. meta::
   :description: Use the ``$project`` aggregation stage to reshape documents for the next pipeline stage by specifying the fields to include, exclude, or compute in the output. Passes documents with fields to next stage in pipeline.

.. contents:: On this page
   :local:
   :backlinks: none
   :depth: 1
   :class: singlecol

Definition
----------

.. pipeline:: $project

   Passes the documents with the requested fields to the
   next stage in the pipeline. The specified fields can be existing
   fields from the input documents or new computed fields.

   ``$project`` can:

   - Include fields from the input documents.
   - Suppress the ``_id`` field.
   - Add new fields or reset the values of existing fields.
   - Exclude fields from the input documents.

Compatibility
-------------

.. |operator-method| replace:: ``$project``

.. include:: /includes/fact-compatibility.rst

Syntax
------

``$project`` has the following form:

.. code-block:: javascript

   { $project: { <specification(s)> } }

The ``$project`` specifications have the following forms:

.. list-table::
   :header-rows: 1
   :widths: 30 70

   * - Form
     - Description

   * - ``<field>: <1 or true>``

     - Specifies inclusion of a field. Non-zero integers are also treated
       as ``true``.

   * - ``_id: <0 or false>``
     - Specifies suppression of the ``_id`` field.

       To exclude a field conditionally, use the :variable:`REMOVE`
       variable instead. For details, see :ref:`remove-var`.

   * - ``<field>: <expression>``

     - Adds a new field or resets the value of an existing field.

       If the expression evaluates to ``$$REMOVE``, the field is
       excluded in the output. For details, see :ref:`remove-var`.

   * - ``<field>: <0 or false>``

     - Specifies exclusion of a field.

       To exclude a field conditionally, use the :variable:`REMOVE`
       variable instead. For details, see :ref:`remove-var`.

       If you specify the exclusion of a field other than ``_id``,
       you **cannot** use any other ``$project``
       specification. This restriction does not apply to
       conditional exclusion of a field using the :variable:`REMOVE`
       variable.

       To exclude fields, you can also use the
       :pipeline:`$unset` stage.

Behavior
--------

Include Fields
~~~~~~~~~~~~~~

- By default, output documents include the ``_id`` field. To include
  other fields from the input documents, specify them explicitly in
  ``$project``.

- If you specify an inclusion of a field that does not exist in the
  document, ``$project`` ignores that field inclusion and
  does not add the field to the document.

_id Field
~~~~~~~~~

By default, the ``_id`` field is included in the output documents.
To exclude the ``_id`` field from the output documents, you
must explicitly suppress the ``_id`` field in
``$project``.

Exclude Fields
~~~~~~~~~~~~~~

If you exclude a field, all other fields appear in the output
documents.

.. code-block:: javascript

   { $project: { "<field1>": 0, "<field2>": 0, ... } } // Return all but the specified fields

If you exclude a field other than ``_id``, you cannot
use other ``$project`` specification forms. If you
exclude fields, you cannot also specify the inclusion of fields, reset
the value of existing fields, or add new fields. This restriction does not
apply to conditional exclusion of a field using the
:variable:`REMOVE` variable.

To exclude fields, you can also use :pipeline:`$unset`.

.. _remove-var:

Exclude Fields Conditionally
~~~~~~~~~~~~~~~~~~~~~~~~~~~~

You can use the variable :variable:`REMOVE` in aggregation expressions
to conditionally suppress a field. For an example, see
:ref:`remove-example`.

Add New Fields or Reset Existing Fields
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

.. note::

   MongoDB also provides :pipeline:`$addFields` to add new fields to
   the documents.

To add a new field or to reset the value of an existing field, specify
the field name and set its value to an expression. For more
information on expressions, see :ref:`aggregation-expressions`.

Literal Values
``````````````

To set a field to a numeric or boolean literal rather than an
expression that resolves to a literal, use the :expression:`$literal`
operator. Otherwise, ``$project`` treats the literal as a flag
for including or excluding the field.

Field Rename
````````````

By specifying a new field and setting its value to the field path of an
existing field, you can effectively rename a field.

New Array Fields
````````````````

The ``$project`` stage supports using the square brackets ``[]``
to directly create new array fields. If you specify array fields that do
not exist in a document, the operation substitutes ``null`` as the value
for that field. For an example, see
:ref:`example-project-new-array-fields`.

.. include:: /includes/project-stage-and-array-index.rst

Embedded Document Fields
~~~~~~~~~~~~~~~~~~~~~~~~

To project, add, or reset a field within an embedded
document, you can use :term:`dot notation`:

.. code-block:: javascript

   "contact.address.country": <1 or 0 or expression>

You can also nest the fields:

.. code-block:: javascript

   contact: { address: { country: <1 or 0 or expression> } }

When you nest fields, you *cannot* use dot notation inside the
embedded document to specify the field. For example, ``contact: {
"address.country": <1 or 0 or expression> }`` is *invalid*.

Path Collision Errors in Embedded Fields
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

You cannot specify both an embedded document and a field within that
embedded document in the same projection.

The following ``$project`` stage fails with a ``Path collision``
error because it attempts to project both the embedded ``contact``
document and the ``contact.address.country`` field:

.. code-block:: javascript
   :copyable: false

   { $project: { contact: 1, "contact.address.country": 1 } }

The error occurs regardless of the order in which you specify the
parent and embedded field. The following ``$project`` fails
with the same error:

.. code-block:: javascript
   :copyable: false

   { $project: { "contact.address.country": 1, contact: 1 } }

$project Stage Placement
~~~~~~~~~~~~~~~~~~~~~~~~

.. |$project| replace:: ``$project``

.. include:: /includes/aggregation/fact-project-stage-placement.rst

Empty Specification
~~~~~~~~~~~~~~~~~~~

MongoDB returns an error if the ``$project`` stage is passed an
empty document.

For example, the following pipeline returns an error:

.. code-block:: javascript

   db.myCollection.aggregate( [ {
      $project: { }
   } ] )
