=============================================================
Install |text-search| and |vector-search| on a Single Cluster
=============================================================

.. facet::
   :name: genre
   :values: tutorial

.. meta::
   :description: Install MongoDB Search and Vector Search on a single Kubernetes cluster using the MongoDB Controllers for Kubernetes Operator. Covers Community and Enterprise editions and replica set topologies.
   :keywords: vector search, text search, mongot, kubernetes, installation,
      replica set, sharded cluster, community, enterprise, external

.. default-domain:: mongodb

.. contents:: On this page
   :local:
   :backlinks: none
   :depth: 2
   :class: singlecol

You can use the |k8s-op-short| to deploy |text-search| and
|vector-search| on a single Kubernetes cluster with your MongoDB
deployment. The ``mongot`` process supports |text-search| and
|vector-search|.

Select your configuration to filter the installation procedure for
your deployment.

.. note::

   Community edition offers a single deployment configuration. If you
   select **Community** for **MongoDB Server**, the other three
   selectors lock. **MongoDB Deployment** is set to **Managed**,
   **Cluster Topology** to **Replica Set**, and **Mongot Instances** to
   **Single**, and you can't change them.

   Overview
   --------

   .. selected-content::

      to run with MongoDB Community Edition v8.2.0 or higher on a
      |k8s| cluster. The ``mongot`` process supports both |text-search|
      and |vector-search|. Optionally, you can enable and configure
      |vector-search| to automatically generate vector embeddings for
      text data in your collections and queries using a supported
      |voyage| embedding model.

      replica set in your |k8s| cluster. The deployment uses |tls|
      certificates to ensure secure communication between MongoDB
      nodes and the ``mongot`` search process.

      About the Managed Envoy Load Balancer
      ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

      When you set ``spec.clusters[].loadBalancer.managed`` in your |fts|
      resource, the operator automatically:

      1. Deploys an Envoy load balancer that distributes traffic across
         the ``mongot`` pods at the application layer (L7). An L7 load
         balancer is required because each ``mongod``-to-``mongot``
         connection is a single, long-lived gRPC stream that an L4 load
         balancer can't distribute across ``mongot`` replicas.
      2. Generates the Envoy configuration, including the listener and
         cluster definitions that route traffic to the ``mongot`` pods.
      3. Creates a Kubernetes Service that exposes the Envoy load
         balancer so the replica set's ``mongod`` processes connect to
         it instead of to the individual ``mongot`` pods.
      4. Manages |tls| by configuring mTLS between |mongod|, the Envoy
         load balancer, and ``mongot``.
      5. Sets the ``mongotHost`` and ``searchIndexManagementHostAndPort``
         search parameters on each ``mongod`` so they point to the Envoy
         load balancer endpoint.

      .. important::

         You don't need to write Envoy configuration, deploy Envoy,
         create the Kubernetes Service manually, or configure |mongod|
         search parameters.

      .. image:: /images/mongodb-k8s-replicaset-search-arch.svg
         :width: 740px
         :alt: Traffic flow diagram showing the Kubernetes cluster
               with an operator-managed MongoDB replica set (rs-0,
               rs-1, rs-2), an Envoy proxy load balancer, and
               ``mongot`` StatefulSet pods.

      About the Managed Envoy Load Balancer
      ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

      When you set ``spec.clusters[].loadBalancer.managed`` in your |fts|
      resource, the operator automatically:

      1. Deploys an Envoy load balancer that distributes traffic to
         each shard's ``mongot`` group at the application layer (L7).
         An L7 load balancer is required because each
         ``mongod``-to-``mongot`` connection is a single, long-lived
         gRPC stream that an L4 load balancer can't distribute across
         ``mongot`` replicas.
      2. Generates the Envoy configuration, including the listener and
         cluster definitions and the |sni|-based routing rules that
         direct each shard's traffic to the correct ``mongot`` group.
      3. Creates a Kubernetes Service for each shard to expose the
         Envoy load balancer to that shard's ``mongod`` processes.
      4. Manages |tls| by configuring mTLS between |mongod|, the Envoy
         load balancer, and ``mongot``.
      5. Sets the ``mongotHost`` and ``searchIndexManagementHostAndPort``
         search parameters on each shard's ``mongod`` so they point to
         the Envoy load balancer endpoint for that shard.

      .. important::

         You don't need to write Envoy configuration, deploy Envoy,
         create the Kubernetes Services manually, or configure |mongod|
         search parameters.

      .. image:: /images/mongodb-k8s-sharded-search-arch.svg
         :alt: Architecture diagram showing the traffic flow from
               operator-managed shards through Envoy Proxy to
               ``mongot`` StatefulSets within a Kubernetes cluster.
         :width: 740px

      About Per-Shard Routing
      ~~~~~~~~~~~~~~~~~~~~~~~~

      In a sharded cluster, each shard has its own data, so |fts|
      deploys a separate ``mongot`` group per shard, and each shard's
      |mongod| must connect to its corresponding ``mongot`` group. Each
      shard's ``mongod`` search parameters point to a server name that
      is unique to that shard. The Envoy load balancer reads the |tls|
      |sni| field from each incoming connection and uses that server
      name to route the request to the ``mongot`` group for the
      matching shard.

      About the Managed Envoy Load Balancer
      ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

      Envoy is a high-performance L7 load balancer. An L7 load balancer
      is required because each ``mongod``-to-``mongot`` connection is a
      single, long-lived gRPC stream. An L4 load balancer can't
      distribute that stream across the ``mongot`` replicas. When you
      set ``spec.clusters[].loadBalancer.managed`` in your
      ``MongoDBSearch`` resource, the |k8s-op-short| automatically:

      1. Deploys an Envoy load balancer that distributes traffic to
         each shard's ``mongot`` group at the application layer (L7).
      2. Generates the Envoy configuration, including the listener and
         cluster definitions and the |sni|-based routing rules that
         direct each shard's traffic to the correct ``mongot`` group.
      3. Creates a Kubernetes Service for each shard to expose the
         Envoy load balancer to that shard's ``mongod`` processes.
      4. Manages |tls| by configuring mTLS between |mongod|, the Envoy
         load balancer, and ``mongot``.

      You don't need to write Envoy configuration, deploy Envoy, or
      create the Kubernetes Services manually. Because your external
      ``mongod`` and ``mongos`` processes aren't operator-managed, you
      set their search parameters to the Envoy load balancer endpoint
      yourself in a later step.

      .. figure:: /images/search_external_sharded_cluster_traffic_flow.png
         :figwidth: 700px
         :alt: Traffic flow for external sharded cluster with managed
               Envoy load balancer

      About Per-Shard Routing
      ~~~~~~~~~~~~~~~~~~~~~~~~

      In a sharded cluster, each shard has its own data, so the
      |k8s-op-short| deploys a separate ``mongot`` group per shard, and
      each shard's ``mongod`` must connect to its corresponding
      ``mongot`` group. Each shard's ``mongod`` search parameters point
      to a server name that is unique to that shard. The Envoy load
      balancer reads the |tls| :abbr:`SNI (Server Name Indication)`
      field from each incoming connection and uses that server name to
      route the request to the ``mongot`` group for the matching shard.

      - A MongoDB Enterprise Edition replica set running version 8.2

         The version format for the Enterprise edition is
         ``X.Y.Z-ent``. Ensure that you choose ``8.2.0`` or later. To
         learn more about MongoDB versioning, see
         :ref:`release-version-numbers` in the MongoDB Manual.

      To learn more about deploying MongoDB Enterprise, see
      :ref:`mdb-resources`.

      - A running MongoDB |com| for managing MongoDB tasks.

      - External DB |tls|

        Your external MongoDB Enterprise deployment must be configured
        to use and require |tls| connections.

      - External DB CA file

        You must have the Certificate Authority (CA) certificate file
        (for example, ``external-ca.crt``) that was used to sign your
        external MongoDB deployment's server certificates.

      - Search Service certificate files

        You must have a valid |tls| server certificate and private key
        (for example, ``search-service.crt``, ``search-service.key``)
        for the ``MongoDBSearch`` service that your external ``mongod``
        processes connect to. The certificate must be valid for the
        hostname of that service (for example,
        ``mdbs-search.example.com``). You store the certificate in a
        secret and reference it from the ``MongoDBSearch`` resource
        through ``spec.security.tls.certsSecretPrefix``, which the
        operator uses to locate the secret. To learn more, see
        :ref:`k8s-fts-vs-settings`.

   Troubleshooting
   ~~~~~~~~~~~~~~~

   .. selected-content::
      :selections: enterprise, k8s-managed, repl, multiple

      Envoy Pod Not Starting
      ``````````````````````

      **Symptoms:** The ``{name}-search-0-lb`` Deployment has 0/1
      ready pods.

      **Check**

      .. code-block:: shell
         :copyable: false

         kubectl describe deployment ${MDB_RESOURCE_NAME}-search-lb-0 -n ${MDB_NS}
         kubectl logs -l app=${MDB_RESOURCE_NAME}-search-lb-0 -n ${MDB_NS}

      **Common causes:**

      - TLS certificate Secrets were not found. Ensure that the
        ``mongot`` and load balancer certificates are created before
        the ``MongoDBSearch`` resource.
      - The ConfigMap ``${MDB_SEARCH_RESOURCE_NAME}-search-lb-0-config``
        does not exist. Verify that the ``MongoDBSearch`` resource
        reached Running state.
      - Image pull errors. Check image pull secrets and registry
        access.

      Search Index Creation Fails
      ````````````````````````````

      **Symptoms:** Search indexes remain in the ``BUILDING`` or
      ``FAILED`` state and do not become queryable.

      **Check**

      .. code-block:: shell
         :copyable: false

         kubectl get pods -n ${MDB_NS} | grep search
         kubectl logs ${MDB_RESOURCE_NAME}-search-0 -n ${MDB_NS}

      **Common causes:**

      - ``mongot`` cannot connect to MongoDB. Verify that the source
        credentials are correct and that the ``MongoDBSearch``
        resource references a valid secret.
      - TLS CA mismatch between |mongod| and ``mongot``. Ensure both
        components are configured to use the same Certificate
        Authority.
      - ``mongot`` pods are not ready yet. Wait for all search pods
        to reach the ``Running`` state before attempting to use
        |text-search| features.

      ``MongoDBSearch`` Stuck in Pending
      ````````````````````````````````````

      **Symptoms:** The ``MongoDBSearch`` resource stays in
      ``Pending`` state.

      **Check**

      .. code-block:: shell
         :copyable: false

         kubectl describe mongodbsearch ${MDB_RESOURCE_NAME} -n ${MDB_NS}
         kubectl get events -n ${MDB_NS} --field-selector involvedObject.name=${MDB_RESOURCE_NAME}

      **Common causes:**

      - The password Secret for the ``search-sync-source`` user was
        not created.
      - TLS certificates are not ready. Run
        ``kubectl get certificates -n ${MDB_NS}`` to check.
      - The cluster is not reachable from the Kubernetes cluster.

      |mongod| Cannot Reach Envoy
      `````````````````````````````

      **Symptoms:** Search queries fail with connection errors on the
      external |mongod| instances.

      **Check:**

      .. code-block:: shell

         # Verify that per-shard proxy Services exist
         kubectl get svc -n ${MDB_NS} | grep proxy-svc

         # From your mongod host, test TLS connectivity to the proxy
         openssl s_client -connect <envoy-endpoint>:27028 -servername <sni-hostname>

      **Common Causes:**

      - The ``MongoDBSearch`` resource is not in ``Running`` state.
        Proxy Services are created only after reconciliation
        completes.
      - Network policies are blocking traffic between the external
        |mongod| hosts and the |k8s| cluster.
      - DNS resolution fails for the proxy Service endpoint. Verify
        that the external |mongod| hosts can resolve the proxy
        Service's external IP or hostname.
      - The ``mongotHost`` parameter on the external |mongod| does
        not match the proxy Service endpoint for that shard.

      Reading ``mongot`` Logs
      ````````````````````````

      Check per-shard ``mongot`` pod logs for connection and
      readiness state:

      .. code-block:: shell

         # Replace <shard-name> with the shard name (for example, shard-0)
         kubectl logs ${MDB_SEARCH_RESOURCE_NAME}-search-0-<shard-name>-0 -n ${MDB_NS} --tail=50

      .. list-table::
         :widths: 35 65
         :header-rows: 1

         * - Issue
           - Log Pattern

         * - Bad credentials or missing user
           - ``MongoSecurityException: Exception authenticating``

         * - TLS CA mismatch or expired cert
           - | ``SSLHandshakeException``
             | ``PKIX path validation failed``

         * - Unreachable host or DNS failure
           - | ``MongoSocketOpenException``
             | ``MongoTimeoutException``

      ``mongot`` logs readiness state initially as follows:

      .. code-block:: shell
         :copyable: false

         Not ready, waiting on replication to be initialized...

      When ``mongot`` is ready, it logs the following:

      .. code-block:: shell
         :copyable: false

         Server is ready!

      To increase log verbosity, set ``spec.logLevel: DEBUG`` on the
      ``MongoDBSearch`` CR.
