
.. _ref-deployment-types:

=====================================
Choose a {+Database-Deployment+} Type
=====================================

.. default-domain:: mongodb

.. contents:: On this page
   :local:
   :backlinks: none
   :depth: 1
   :class: singlecol

|service-fullname| can deploy two types of cloud databases: {+serverless-instances+}
and {+clusters+}.

- **{+Serverless-Instances+}** require minimal configuration. |service|
  automatically scales the storage capacity, storage throughput, and computing power for a
  {+serverless-instance+} seamlessly to meet your workload requirements. They
  always run the latest :manual:`MongoDB version </reference/versioning>`, and
  you :ref:`only pay for the operations that you run <serverless-instance-costs>`.

- **{+Clusters+}** give you more flexibility in choosing your database configuration.
  You can set the :doc:`cluster tier </cluster-tier>`, use advanced capabilities
  such as :manual:`sharding </sharding>` and :doc:`Continuous Cloud Backups
  </backup/cloud-backup/overview>`, distribute your data to :doc:`multiple
  regions and cloud providers </cluster-config/multi-cloud-distribution>`, and :doc:`scale
  your {+cluster+} </scale-cluster>` on-demand. You can also enable :doc:`autoscaling
  </cluster-autoscaling>`, but it requires preconfiguration. MongoDB bills {+clusters+} based on
  the deployment configuration and :doc:`cluster tier </cluster-tier>`.

.. include:: /includes/note-serverless-in-preview.rst

You can use both {+clusters+} and {+serverless-instances+} in the same |service|
project.

Use Cases
---------

The best {+database-deployment+} for you depends on your use case and feature requirements.

Create a {+serverless-instance+} if you want to:
  
- Get started quickly with minimal database configuration.
- Have your database scale automatically and dynamically to meet your workload.
- Run infrequent or sparse workloads.
- Develop or test in a cloud environment.

Create a {+cluster+} if you want to:
  
- Choose a specific database configuration based on your workload requirements.
- Define database :doc:`scaling behavior </cluster-autoscaling>`.
- Run high-throughput production workloads.
- Always have capacity available.
