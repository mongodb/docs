# Compatibility and Requirements for `mongot`

Draft status: internal working draft for documentation planning. This version intentionally blends currently published support statements with planned GA guidance so the page can be reviewed as a single customer-facing concept before final publication.

This page describes version compatibility, minimum component requirements, operating system support, and deployment compatibility for self-managed MongoDB Search and Vector Search.

On this page, MongoDB Search Community Edition refers to the self-managed Linux binary or container distribution used with MongoDB Community Edition. MongoDB Search for Enterprise Advanced refers to MongoDB Search deployed and managed through MongoDB Controllers for Kubernetes Operator (MCK) for use with MongoDB Enterprise Advanced.

Unless otherwise noted, Community Edition statements apply to `mongot` 1.70.0 and later.

## Search and Vector Search Compatibility and Requirements

This section describes the version compatibility and minimum component requirements for running MongoDB Search and Vector Search with MongoDB Controllers for Kubernetes Operator.

Use it to ensure that your planned deployment meets the requirements before you deploy `mongot`.

### Compatibility

#### `mongot` and `mongod`

For MCK-managed deployments, any supported `mongot` version is compatible with any supported `mongod` version of MongoDB Server 8.3 or later.

Concretely:

* `mongot` 1.70.0 works with `mongod` 8.3.x.
* Features added in newer `mongot` might require a newer `mongod` to function.
* Existing features keep working when `mongot` is upgraded without upgrading `mongod`, and vice versa.

This allows you to upgrade `mongot` and `mongod` independently rather than coordinating both in the same maintenance window.

#### `mongot` and the Kubernetes Operator

The Kubernetes Operator declares the `mongot` versions it supports. Use the Kubernetes Operator version that ships alongside or after the `mongot` version you intend to deploy. The compatibility table later on this page starts with `mongot` 1.70.0 and MCK 1.9.0, which are the minimum documented versions for MCK-managed Search and Vector Search deployments. To learn more, see [MongoDB Controllers for Kubernetes Operator Compatibility](https://www.mongodb.com/docs/kubernetes/upcoming/tutorial/plan-k8s-op-compatibility#std-label-k8s-compatibility).

#### Supported Platforms and Editions

MongoDB Search and Vector Search on Kubernetes inherit the platform support of MongoDB Controllers for Kubernetes Operator. Confirm that your cluster meets the Kubernetes Operator requirements before you deploy `mongot`:

* For supported MongoDB Server versions and base images, see [Supported Platforms and MongoDB Versions](https://www.mongodb.com/docs/kubernetes/upcoming/tutorial/plan-k8s-op-compatibility#std-label-k8s-mdb-version-compatibility).
* For supported hardware architectures, see [Supported Hardware Architectures](https://www.mongodb.com/docs/kubernetes/upcoming/tutorial/plan-k8s-op-compatibility#std-label-k8s-supported-hardware-arch-compatibility).
* For supported Ops Manager versions, see [Cloud Manager and Ops Manager Versions](https://www.mongodb.com/docs/kubernetes/upcoming/tutorial/plan-k8s-op-compatibility#std-label-k8s-onprem-version-compatibility).

You can run MongoDB Search and Vector Search against either edition of MongoDB Server that the Kubernetes Operator manages:

* Community Edition: manage the source `mongod` through the `MongoDBCommunity` resource. Ops Manager and Cloud Manager do not apply in this configuration.
* Enterprise Edition: manage the source `mongod` through the `MongoDB` resource, paired with Ops Manager or Cloud Manager.

Both editions use the same `MongoDBSearch` resource to deploy `mongot`. For a deployment that is external to the Kubernetes Operator, reference it through `spec.source.external` on the `MongoDBSearch` resource. To learn more, see [MongoDB Search and Vector Search Settings](https://www.mongodb.com/docs/kubernetes/upcoming/reference/fts-vs-settings#std-label-k8s-fts-vs-settings).

## Compatibility Notes

### Load Balancing for Multi-mongot Topologies

If a replica set or shard uses more than one `mongot` instance, use an L7 load balancer. An L4 load balancer is not sufficient because `mongod` opens a long-lived connection to `mongot` and an L4 balancer cannot distribute traffic at the gRPC stream level.

### Sharded Deployments

For sharded deployments, each shard has its own `mongot` group. Shards do not share `mongot` instances. In Kubernetes-based deployments, MCK can automate this topology. In manually orchestrated deployments, customers must create and manage the per-shard topology themselves.

### Enterprise Production

For Enterprise Advanced, the intended production posture is MongoDB managed on VMs, bare metal, or Kubernetes, with `mongot` managed by MongoDB Controllers for Kubernetes Operator.

### Ops Manager

Ops Manager can manage the MongoDB deployment used by Search and can surface curated MongoDB Search visibility, but it is not the management plane for MongoDB Search itself.

### Docker

Docker should be treated as two separate cases:

* `atlas-local` is for local development and continuous integration only and does not pair with an external MongoDB deployment.
* Linux containerized MongoDB Search Community Edition can be a packaging and runtime mechanism on supported Linux platforms.
* Enterprise production support still assumes MCK-managed MongoDB Search for Enterprise Advanced rather than standalone container lifecycle management.

### Packaging

For MongoDB Search Community Edition at GA, tarball and container installs are the expected packaging paths. Native package-manager installation with DEB or RPM is planned later and should not be implied as available on day one.
