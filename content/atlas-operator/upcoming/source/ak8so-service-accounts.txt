=====================================================
Configure Access to |service| Using Service Accounts
=====================================================

.. meta::
   :description: Configure Atlas Kubernetes Operator access to Atlas using Atlas API Service Accounts instead of Programmatic API Keys.

.. default-domain:: mongodb

.. contents:: On this page
   :local:
   :backlinks: none
   :depth: 2
   :class: singlecol

Overview
--------

|service| API Service Accounts provide a secure, token-based
alternative to :ref:`Programmatic API Keys <about-org-api-keys>` for
authenticating |ak8so| with the {+atlas-admin-api+}. Service Accounts
use short-lived access tokens derived from a ``clientId`` and
``clientSecret`` pair, reducing the risk of long-lived credential
exposure.

MongoDB recommends Service Accounts over Programmatic API Keys for
new deployments. Service Accounts offer the following benefits:

- Token-based authentication with automatic token refresh managed by 
  the operator.
- Support for both organization-level and project-level scopes.
- Auditable, distinct identity for your |ak8so| workloads.

.. important:: Service Account Secret Expiration

   Unlike API Keys, Service Account secrets are
   **expirable**. You can set a :abbr:`TTL (Time to Live)` of up to 
   one year maximum. You must rotate Service Account secrets before 
   they expire to maintain uninterrupted access. |service| 
   automatically raises a **Service Account Secrets are about to 
   expire** alert as the expiry date approaches. Plan your rotation 
   schedule accordingly.

   This applies to both organization-level and project-level Service
   Accounts.

Prerequisites
-------------

Before you begin, ensure you have the following:

- An |service| account with the **Organization Owner** role for
  organization-level Service Accounts, or **Project Owner** role for
  project-level Service Accounts.
- A running |k8s| {+cluster+} with |ak8so| installed.
- ``kubectl`` access to the |k8s| {+cluster+} where |ak8so| runs.

Procedure
---------

.. _ak8so-sa-create:

.. procedure::
   :style: normal

   .. step:: Create the Service Account in |service|

      Create a Service Account in |service| using the {+atlas-ui+} or the
      {+atlas-admin-api+}. To learn how to create an organization-level 
      Service Account, see `Create a Service Account in an 
      Organization <https://www.mongodb.com/docs/atlas/configure-api-access/?interface=atlas-ui&programmatic-access=service-account#create-a-service-account-for-an-organization>`__.
      To learn how to create a project-level Service Account, see 
      `Create a Service Account in a Project 
      <https://www.mongodb.com/docs/atlas/configure-api-access/?interface=atlas-ui&programmatic-access=service-account#create-a-service-account-in-a-project>`__.

      Copy the ``clientId`` and ``clientSecret``. Store them securely
      because you cannot retrieve the ``clientSecret`` again after
      creation.

      .. _ak8so-sa-secret:

   .. step:: Create the |k8s| Connection Secret

      Create a |k8s| |k8s-secret| containing the ``orgId``, ``clientId``,
      and ``clientSecret``.

      - For a ``global`` |ak8so| |k8s-secret|, run the following commands:

        .. note::

           The name of the ``global`` |ak8so| secret must conform to
           the predefined format. The default name of the |ak8so| deployment
           is ``mongodb-atlas-operator``. So, the |k8s-secret| should be
           named ``mongodb-atlas-operator-service-account``.

        .. code-block:: sh

           kubectl create secret generic mongodb-atlas-operator-service-account \
              --from-literal="orgId=<the_atlas_organization_id>" \
              --from-literal="clientId=<the_service_account_client_id>" \
              --from-literal="clientSecret=<the_service_account_client_secret>" \
              -n <operator_namespace>

           kubectl label secret mongodb-atlas-operator-service-account \
              atlas.mongodb.com/type=credentials \
              -n mongodb-atlas-system

      - For a ``project`` |ak8so| |k8s-secret|, run the following commands:

        .. code-block:: sh

           kubectl create secret generic my-project-connection \
              --from-literal="orgId=<the_atlas_organization_id>" \
              --from-literal="clientId=<the_service_account_client_id>" \
              --from-literal="clientSecret=<the_service_account_client_secret>" \
              -n <atlas_project_namespace>

           kubectl label secret my-project-connection \
              atlas.mongodb.com/type=credentials \
              -n <atlas_project_namespace>

      .. _ak8so-sa-reference:

   .. step:: Reference the Secret from ``AtlasProject``

      Reference the |k8s-secret| from your ``AtlasProject`` custom resource
      using :ref:`spec.connectionSecretRef.name
      <atlasproject-spec-connectionsecretref>`.

      If you created a ``global`` |k8s-secret|, all projects use it by
      default and no additional configuration is needed.

      If you created a ``project`` |k8s-secret|, add the
      ``connectionSecretRef`` to your ``AtlasProject`` custom resource:

      .. code-block:: yaml

         apiVersion: atlas.mongodb.com/v1
         kind: AtlasProject
         metadata:
           name: my-project
         spec:
           name: "My Project"
           connectionSecretRef:
             name: my-project-connection

      .. _ak8so-sa-verify:

   .. step:: Verify the Operator Uses the Service Account

      After you configure the |k8s-secret|, verify that |ak8so| 
      authenticates successfully:

      .. code-block:: sh

         kubectl get atlasproject my-project -o jsonpath='{.status.conditions}'

      Check that the ``Ready`` condition is ``True``. If |ak8so| can't
      authenticate with the Service Account credentials, the conditions
      show an error describing the issue.

YAML Example
-------------

The following example shows a Connection Secret configured for a 
Service Account:

.. code-block:: yaml

   apiVersion: v1
   kind: Secret
   metadata:
     name: my-project-connection
     namespace: atlas-operator
     labels:
       atlas.mongodb.com/type: credentials
   type: Opaque
   stringData:
     orgId: "5d39f0719ccf646554345678"
     clientId: "mdb_sa_id_67890abcdef1234567890abc"
     clientSecret: "mdb_sa_sk_abcdef1234567890abcdef12"

Considerations
--------------

Token Management
~~~~~~~~~~~~~~~~

|ak8so| manages access tokens automatically. When the |ak8so| detects
a Service Account Connection Secret, it obtains an access token from
the {+atlas-admin-api+} and stores it in a separate |k8s| |k8s-secret|.
Do **not** manually edit or delete the Access Token Secret.
|ak8so| refreshes tokens as needed.

Credential Rotation
~~~~~~~~~~~~~~~~~~~

When you rotate the Service Account secret in |service|:

1. Update the |k8s| Connection Secret with the new ``clientSecret``.
#. The |ak8so| detects the change and acquires a new access token on
   the next reconcile cycle. No restart is required.

.. important::

   Service Account secrets have a maximum :abbr:`TTL (Time to Live)` 
   of one year. Monitor the **Service Account Secrets are about to 
   expire** alert in |service| and rotate your secrets before they 
   expire to avoid authentication failures. This applies to both 
   organization-level and project-level Service Accounts.

IP Access List
~~~~~~~~~~~~~~

The Service Account's |api| access list must include the egress
:abbr:`IP (Internet Protocol)` address or :abbr:`CIDR (Classless
Inter-Domain Routing)` block used by the |ak8so| pod, the same as
for :ref:`Programmatic API Keys <about-org-api-keys>`. If the
operator pod's egress IP is not on the Service Account access list,
|api| requests fail.
