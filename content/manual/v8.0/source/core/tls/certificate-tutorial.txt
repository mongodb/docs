==============================
Obtain TLS Server Certificates
==============================

.. meta::
   :description: Obtain TLS server certificates from a public or private certificate authority to enable encrypted connections for each node in your MongoDB replica set deployment.

.. contents:: On this page
   :local:
   :backlinks: none
   :depth: 1
   :class: singlecol

Obtain server certificates to enable |tls| encryption for your self-managed 
MongoDB replica set deployments.

.. include:: /includes/tls/note-self-managed-only.rst

Before you Begin
----------------

Before you start, ensure you have the following information and resources:

- You have a self-managed MongoDB replica set deployment that you want to 
  secure with TLS.
- You have at least one admin user enabled on your deployment 
  to verify TLS connections in later tutorials. If you want to enable X.509
  client authentication, the admin user must have
  at least the :authrole:`userAdmin` role to create and modify users in
  the ``$external`` database.
- You have a hostname for each node in your deployment, such as ``mongo0.example.com``,
  ``mongo1.example.com``, and ``mongo2.example.com``. If you are using a public 
  :abbr:`CA (Certificate Authority)`, you must have a registered domain name that
  corresponds to these hostnames.
- You have `OpenSSL <https://www.openssl.org/>`__ installed on your machine.
- If you are planning on using a public CA, 
  such as Let's Encrypt or DigiCert, you know which public CA you are using.
  If you are planning on using a private CA, 
  you have access to your organization's :abbr:`PKI (Public Key Infrastructure)` 
  information.
  The process for obtaining certificates might be different based
  on the CA you use. However, you must end with the same ``.pem`` files described
  in the :ref:`final result <tls-certificate-final-state>` section of this tutorial.
- You have your preferred command line interface open.
- You know your deployment TLS configuration requirements and whether your certificates
  need ``clientAuth`` :abbr:`EKU (Extended Key Usage)` based on the TLS Planning page.
   
Steps
-----

This tutorial creates one certificate called ``mongo0.pem`` for the first node
in your deployment. When generating certificates for additional nodes, be specific
in your file names. For example, name the certificate for your first secondary node
``mongo1.pem``.

.. composable-tutorial::
   :options: certificate-type
   :defaults: public-ca

   .. selected-content::
      :selections: public-ca

      Follow this tutorial to obtain a server certificate from a public CA. 
      **You must obtain a certificate for each node in your
      deployment.**

      .. procedure::
         :style: normal

         .. step:: Prepare your workspace

            .. include:: /includes/tls/step-prepare-workspace.rst

         .. step:: Generate a private key

            .. note::

               If you are using a CA that uses the ACME protocol like Let's Encrypt, 
               the CA generates the certificate for you. In this case,
               follow your CA's instructions instead and resume this tutorial at
               the step where you create ``.pem`` files from your certificate
               and private key.

            .. include:: /includes/tls/step-gen-private-key.rst

         .. step:: Create a certificate signing request

            .. important::

               Starting in April 2026, major public CAs stopped issuing TLS 
               certificates that support client authentication. If you are using
               a public CA, do not include ``extendedKeyUsage = clientAuth``
               in your :abbr:`CSR (Certificate Signing Request)`.
               If you need server certificates for client authentication, use a private CA.
               See the `Public Certificate Authority Policy Changes Affecting mTLS <https://www.mongodb.com/resources/products/alerts/public-certificate-authority-policy-changes-affecting-mtls/>`__
               Technical Advisory for more information.

            .. include:: /includes/tls/step-create-csr.rst

         .. step:: Submit the CSR to a public CA

            Sign in to your chosen public CA's portal and follow their instructions 
            to submit your CSR. When prompted, upload or paste the contents of your 
            ``mongo0.csr`` file. Choose the appropriate validation level:

            - **Domain Validation (DV)**: The CA verifies that you control the domain 
              for which you're requesting a certificate. This is typically the 
              quickest and least expensive option.

            - **Organization/Extended Validation (OV/EV)**: Choose this if your organization
              requires a higher level of trust.

            Follow the CA's instructions to complete the validation process. After successful
            completion, the CA issues your certificate.

         .. step:: Download and verify the certificate and chain
            
            .. include:: /includes/tls/step-download-certificate.rst

            .. include:: /includes/tls/step-verify-certificate.rst

            .. include:: /includes/tls/step-verify-ca-bundle.rst

         .. step:: Create ``.pem`` files

            .. include:: /includes/tls/step-combine-files.rst

         .. step:: Organize the certificate and chain files

            .. include:: /includes/tls/step-copy-files.rst

         .. step:: Repeat for other nodes in deployment

            .. include:: /includes/tls/step-repeat-for-nodes.rst
            
            .. include:: /includes/tls/note-one-cert-bundle.rst

   .. selected-content::
      :selections: private-ca

      If your organization operates its own internal PKI or private CA, obtain 
      certificates from that CA. **You must obtain a certificate for each node in your
      deployment.**

      .. procedure::
         :style: normal

         .. step:: Prepare your workspace

            .. include:: /includes/tls/step-prepare-workspace.rst

         .. step:: Review your organization's PKI requirements

            Check your internal :abbr:`PKI (Public Key Infrastructure)` 
            documentation, or contact your security team, for information like
            required key sizes, algorithms, or :abbr:`SANs (Subject Alternative Names)`.

         .. step:: Generate a private key

            .. note::

               Your CA might use a different process for certificate generation,
               such as a web interface. In this case,
               follow your CA's instructions instead and resume this tutorial at
               the step where you create ``.pem`` files from your certificate
               and private key.

            .. include:: /includes/tls/step-gen-private-key.rst

         .. step:: Create a certificate signing request

            Ensure that your :abbr:`CSR (Certificate Signing Request)` meets 
            your organization's PKI policies.

            .. include:: /includes/tls/step-eku-note.rst

            .. include:: /includes/tls/step-create-csr.rst

         .. step:: Submit the CSR to your private CA

            Upload ``mongo0.csr`` to your internal PKI portal, or send it to your
            PKI administrators as instructed. 

         .. step:: Download and verify the certificate and chain
            
            .. include:: /includes/tls/step-download-certificate.rst

            .. include:: /includes/tls/step-verify-certificate.rst

            .. include:: /includes/tls/step-verify-ca-bundle.rst

         .. step:: Create ``.pem`` files

            .. include:: /includes/tls/step-combine-files.rst

         .. step:: Organize the certificate and chain files

            .. include:: /includes/tls/step-copy-files.rst
         
         .. step:: Repeat for other nodes in deployment

            .. include:: /includes/tls/step-repeat-for-nodes.rst

            .. include:: /includes/tls/note-one-cert-bundle.rst

   .. selected-content::
      :selections: self-signed

      Use self-signed certificates for local development and
      testing. **Do not use self-signed certificates in production.**

      .. procedure::
         :style: normal

         .. step:: Prepare your workspace

            .. include:: /includes/tls/step-prepare-workspace.rst

         .. step:: Generate a private key

            .. include:: /includes/tls/step-gen-private-key.rst

         .. step:: Create a configuration file

            .. include:: /includes/tls/step-eku-note.rst

            Create a minimal OpenSSL configuration file called ``csr.conf`` that uses the
            following format:

            .. literalinclude:: /includes/tls/csr-config-template.rst

            For more information about the fields in this file, see the OpenSSL 
            `config file documentation <https://docs.openssl.org/3.6/man5/config/>`__.
            Ensure that you fill out both the Common Name (``CN``) and Subject Alternative
            Name (``alt_names``) fields in the configuration file.
                        
         .. step:: Generate a self-signed certificate

            Generate a certificate with the private key and configuration file
            that you created:

            .. code-block:: bash

               openssl req -new \
                  -x509 -days 365 \
                  -key mongo0.key \
                  -out mongo0.crt \
                  -config csr.conf

            This command creates a self-signed 
            certificate, ``mongo0.crt``, that is valid for 365 days.
             
         .. step:: Verify the certificate

            .. include:: /includes/tls/step-verify-certificate.rst

         .. step:: Create ``.pem`` files

            .. include:: /includes/tls/step-combine-files.rst

            You can use your self-signed certificate as the CA file.
            For consistency with this tutorial, rename the certificate:

            .. code-block:: bash

               cp mongo0.crt ca.pem

         .. step:: Organize the certificate and chain files

            .. include:: /includes/tls/step-copy-files.rst

         .. step:: Repeat for other nodes in deployment

            .. include:: /includes/tls/step-repeat-for-nodes.rst

Final Result
------------

At the end of this tutorial, you have the following ``.pem`` files 
in ``/etc/ssl/mongodb``:

- For **each node**, you have a ``.pem`` file that contains the certificate and 
  private key for that node, such as ``mongo0.pem``, ``mongo1.pem``, and ``mongo2.pem``.

- For the deployment overall, you have the intermediate CA certificate that issued
  the certificates for each node, such as ``ca.pem``.
