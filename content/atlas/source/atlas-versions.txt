.. _atlas-versions:

=========================
MongoDB Versions in Atlas
=========================

.. default-domain:: mongodb

.. facet::
   :name: genre
   :values: reference

.. meta:: 
   :keywords: Atlas versions, mongodb versioning, version management, major versions, latest version
   :description: Learn about MongoDB version options for Atlas clusters, including Latest Version With Auto Upgrades. Compare availability by cluster type and understand how to select or change your cluster version.

.. contents:: On this page
   :local:
   :backlinks: none
   :depth: 2
   :class: singlecol

The `MongoDB Software Lifecycle Schedules
<https://www.mongodb.com/legal/support-policy/lifecycles/>`__ page
describes the supported versions and end-of-life dates for MongoDB Server releases. Depending on your 
{+cluster+} type, |service| offers different version-management options. 
For {+Dedicated-clusters+}, |service| provides two release cadence options: major versions, 
or ongoing updates using :guilabel:`Latest Version With Auto Upgrades`. 
Free and Flex clusters don't let you choose a release cadence; they stay on a major version (currently, MongoDB 8.0), 
and |service| automatically upgrades them.

For most |service| deployments, :guilabel:`Latest Version With Auto Upgrades` is the best 
way to receive the latest MongoDB features and enhancements automatically as |service| rolls them out.
However, you can opt for tighter version control and choose to stay on a specific major version.

To learn more about MongoDB Server versions in general, see :manual:`MongoDB Versioning </reference/versioning/>`.


.. _latest_version_with_auto_upgrades:

Latest Version with Auto Upgrades
---------------------------------

When you :ref:`choose your cluster version <create-cluster-version>` and select 
:guilabel:`Latest Version With Auto Upgrades`, |service| automatically upgrades 
your cluster to the latest available MongoDB major or minor version, enables 
new features as they become available, and upgrades patch releases through a rolling 
process to maintain cluster availability.

In most cases, we recommend opting in to the automatic version upgrade. It means |service|
keeps your cluster current, and saves you the need to track the MongoDB release schedule. 
When choosing :guilabel:`Latest Version With Auto Upgrades`, you can't select a specific 
major release, minor release, or patch version. 

Switching from Auto Upgrades to a Major Version
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
If your cluster is on a minor version while on :guilabel:`Latest Version With Auto Upgrades`, 
you can switch to a major-version cadence only when the next major version becomes available.

For example, if you opt in while MongoDB 8.2 is the latest release, your cluster remains 
on the latest minor version until MongoDB 9.0 is released. You can switch back to a major 
version cadence only in the window between the 9.0 release and the first 9.x minor release. If you 
miss that window, you must wait until the next major release cycle.

.. include:: /includes/mongosync-minor-version-compatibility-restriction.rst

Major Version
-------------
If you need to remain on a stable release and maintain control of major-version
upgrades, you can choose to remain on a major version. You can
:ref:`upgrade the major version <upgrade-major-MongoDB-version>`
of your |service| cluster at any time by :ref:`modifying the cluster <scale-cluster>`.
Major versions also automatically receive patch release updates.

Before upgrading a major version in production, test your application
against a staging cluster and review compatibility changes in the
:ref:`release notes <server-release-notes-landing>`. If applicable,
rely on the :ref:`Stable API <stable-api>`.

End of Life for a Major Version
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
|service| sends you an email notification at least six months before
a MongoDB version reaches :ref:`end of life <atlas-eol-upgrade>`. You can
also find the information on the `MongoDB Software Lifecycle Schedules
<https://www.mongodb.com/legal/support-policy/lifecycles/>`__ page.
After the cut-off date, |service| upgrades your clusters to the next
MongoDB version unless you request and receive approval for an
extension. Upgrade your cluster before the cut-off date to ensure
that your services and applications experience no downtime or
compatibility issues with the new MongoDB version.

Version Availability
--------------------

The following table shows the MongoDB versions available for each
{+cluster+} type.

.. list-table::
   :header-rows: 1

   * - Version
     - Available on ``M10+``
     - Available on ``M0`` and {+Flex-clusters+}

   * - MongoDB 7.0
     - :icon:`check-square`
     -

   * - MongoDB 8.0
     - :icon:`check-square`
     - :icon:`check-square`

   * - MongoDB 8.3 * 
     - :icon:`check-square`
     -

* Version 8.3 is available only for clusters with :guilabel:`Latest Version With Auto Upgrades`.  

.. important::

   If your project contains a :ref:`custom role <mongodb-roles>`
   that uses actions introduced in a specific MongoDB version, you
   can't create a {+cluster+} with a MongoDB version earlier than
   that version unless you delete the custom role.

.. _scale-cluster-version:

Select or Change the Cluster Version
------------------------------------

.. include:: /includes/fact-upgrade-best-practices.rst

To select or change the cluster version:

.. procedure::
   :style: normal

   .. step:: Open the cluster configuration.

      - To set the version for a new cluster, complete the steps to
        :ref:`create a cluster <create-new-cluster>` and expand
        :guilabel:`Additional Settings`.
      - To change the version for an existing cluster, click
        :guilabel:`Edit Config` to open the cluster form and
        expand :guilabel:`Additional Settings`.

   .. step:: Select a version.

      From the :guilabel:`Select a version` dropdown, select the
      version for your cluster.

      |service| supports the following upgrade paths:

      - MongoDB 7.0 to MongoDB 8.0
      - MongoDB 8.0 to Latest Version With Auto Upgrades
